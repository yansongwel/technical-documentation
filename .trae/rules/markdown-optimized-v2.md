【核心铁律】
- **没有验证的文档禁止输出**。每一个部署文档都必须独立经过完整的伪集群验证流程。
- **未验证的内容严禁写入文档**。监控下载、备份恢复、Exporter 采集、日志清理、配置参数、目录结构等，
  凡是文档中出现的内容，必须在伪集群中实际验证通过后才可编写。
- **一个文档只讲一种方案**。多种集群方案或多个主流版本不得堆砌在同一文档中。
- 验证是手段，不是目的。伪集群仅用于确认正确性，最终文档输出的是生产级多节点集群方案。

【可读性铁律（强制）】
- **每一段命令块之前必须写清楚：当前步骤作用（可用加粗或引用简要说明，避免冗长堆砌）**。禁止“只给一堆命令让读者盲目执行”。
- **每一段命令块之后必须给出至少 1 个验证点**（命令 + 预期输出关键字/状态），让执行者知道是否成功。
- **多节点/多角色步骤必须显式写出差异**：例如 Router-01 vs Router-02、Master vs Replica、Primary vs Secondary；并用“必须相同/必须不同”的方式标注关键字段。
- **配置文件必须逐行注释到位**：每个关键参数都要解释“作用 + 为什么推荐/为什么必须 + 哪些环境要改”，并在配置块上方给出“必须修改项清单（表格）”。

【强制写法模板（推荐直接复制）】

每个“步骤小节”**必须**包含以下结构（可用表格或短段落）；**禁止**只给命令而无说明，让读者盲目执行：

1) **简要说明**：命令块前用**加粗**或 `>` 引用，一句话点明本步作用即可，避免每步都写「本步目的」「本步做了什么」等冗长模板，保持专业简洁
2) **执行命令**：给出命令块（bash）
3) **验证方法**：给出验证命令 + 预期输出关键字（例如 `Active: active (running)` / `ONLINE` / `PONG`）
4) **失败常见原因（推荐）**：列出 2~3 条最常见踩坑点（端口未通、参数没改、权限不对），便于读者自查

> ⚠️ **防偷懒约束（强制）**：
> - 第零步确定 N 个文档需要生成，则必须执行 N 轮独立的验证-清理-写文档循环
> - 禁止"验证了 Cluster 就直接凭记忆写 Sentinel"——两种方案的配置文件、启动参数、
>   管理命令、集群创建方式完全不同，必须各自独立验证
> - 禁止"验证了 7.x 就直接在文档中把版本号改成 8.x"——不同大版本的配置参数名称、
>   默认值、新增/废弃特性可能完全不同，必须各自独立验证
> - 每个方案的验证目录用不同名称区分：`/tmp/{service}-{方案名}-verify/`
>   例如 `/tmp/redis-cluster-verify/`、`/tmp/redis-sentinel-verify/`

【一个文档只讲一种方案】

当一个中间件存在多种生产级集群方案或多个主流版本时，禁止将所有方案/版本堆砌在同一个文档中。
必须拆分为多个独立文档，每个文档只聚焦一种方案，确保清晰、无干扰。

**拆分触发条件（满足任一条即拆分）**：
  1. **多种集群架构**：中间件提供多种官方支持的生产集群模式
     - 示例：Redis（Cluster 分片模式 vs Sentinel 哨兵模式）
     - 示例：MySQL（MGR 组复制 vs 主从复制 vs InnoDB Cluster）
     - 示例：Kafka（KRaft 模式 vs ZooKeeper 模式）
     - 示例：Elasticsearch（Hot-Warm-Cold 架构 vs 常规集群）
  2. **主流版本差异大**：两个大版本之间存在架构性变更、配置不兼容、或客户端/SDK 需要改造
     - 判断标准：如果从版本 A 升级到版本 B 需要修改业务代码或客户端配置，则视为"差异大"
     - 示例：Redis 7.x vs 8.x、Kafka 2.x(ZK) vs 3.x+(KRaft)、MySQL 5.7 vs 8.0
  3. **部署方式差异大**：同一中间件的原生部署与 K8s Operator/Helm 部署流程完全不同时
     - 示例：Ceph 原生部署 vs Rook（K8s Operator 编排 Ceph）
     - 示例：Elasticsearch 原生集群 vs ECK（Elastic Cloud on Kubernetes）

**不需要拆分的情况（仅一种方案 + 无版本分歧）**：
  - 单机类服务：Grafana、Jenkins、Nginx（仅一种部署方式，无需选型文档）
  - 仅一种集群方案：etcd（只有 Raft 集群）、NATS（只有原生 Cluster）
  - 版本间无破坏性变更：直接使用最新稳定版，不需要双版本文档

【版本选择原则】

- **不盲目追新**：优先选择大厂广泛采用、社区成熟、生态完善的版本
- **生产稳定性优先**：若最新大版本发布不足 6 个月且无大厂背书，推荐上一个 LTS/稳定版
- **业务代码兼容性**：若新版本要求客户端 SDK 升级或业务代码改动，必须在选型文档中明确标注
- **双版本策略**：当两个大版本都有大量生产用户时（如 Redis 7.x 和 8.x），两个版本都提供部署文档
- 版本决策建议写在选型总览文档中，各部署文档的第 2 章仅引用总览文档，不重复

【验证方式选择（根据中间件类型）】

并非所有中间件都适合 Docker 伪集群验证，根据类型选择合适的验证方式：

| 验证方式 | 适用场景 | 示例 |
|---------|---------|------|
| **Docker Compose 伪集群** | 数据库、消息队列、配置中心、缓存等有状态服务 | Redis、MySQL、Kafka、etcd、MongoDB |
| **Docker 单容器** | 无状态单机服务、Web 应用 | Nginx、Grafana、Jenkins、Harbor |
| **宿主机直接安装** | 语言运行环境、系统级工具、内核级服务 | Python/Go/Java、iptables/firewalld、LVS |
| **宿主机命令验证** | IaC/配置管理工具（工具本身不需要集群） | Ansible、Terraform、SaltStack、Vagrant |
| **Kind/K3s 轻量集群** | K8s 附属组件（需要 K8s API） | Helm、Kustomize、Velero、Cert-Manager、Istio |
| **跳过集群验证，仅验证命令/配置** | K8s 集群部署工具（本身就是创建集群的） | kubeadm、kubeasz、kubespray |

- 无论使用哪种验证方式，**版本探测 + 下载地址验证 + 命令存在性验证**都是必须的
- 若无法在当前环境启动完整集群，至少验证：安装包可下载、二进制可执行、配置文件语法正确、
  关键命令 `--help` 可用，并在文档中标注"以下预期输出基于 {验证环境} 确认"

**验证目的**：在本机通过 Docker / Docker Compose 伪集群验证以下内容的正确性——
  - 下载地址是否可达
  - 安装命令是否存在且可执行
  - 配置参数是否正确且生效
  - 管理/运维命令是否可用且输出符合预期

**验证是手段，不是目的**：
  - 伪集群仅用于验证上述内容的正确性，验证通过后立即清理环境
  - 最终文档输出的是**生产级多节点集群方案**，而不是 Docker 伪集群方案
  - 伪集群的 Docker Compose 文件可作为第 6 章"快速体验部署"的内容保留

【Docker 伪集群验证规范】
- 验证环境：在当前 Linux 服务器上使用 Docker / Docker Compose 启动伪集群
- 验证目录按方案区分：`/tmp/{service}-{方案名}-verify/`
  例如 `/tmp/redis-cluster-verify/`、`/tmp/redis-sentinel-verify/`
- 伪集群最低节点要求（仅用于验证，不写入生产文档主体）：
  - 数据库类（Redis/MySQL/MongoDB/ES）：最少 3 节点，验证集群选举与主从同步
  - 消息队列（Kafka/RabbitMQ/RocketMQ）：最少 3 Broker + 必要依赖组件
  - 配置中心（Nacos/Consul/etcd/ZooKeeper）：最少 3 节点，验证 Raft/ZAB 选举
  - 单机类服务（Nginx/Jenkins/Grafana）：单容器启动即可
- **验证内容必须全面覆盖**（★ 缺一不可，未验证项禁止写入文档）。
  每项验证均有编号（#1~#14），子步骤详解中已标注各子步骤覆盖的验证项编号，确保无遗漏：

  2. **容器/服务正常启动**：无报错退出，`docker ps` 确认全部 running
  3. **集群节点互通与状态检查**：如 `cluster info`、`cluster nodes`、`show replicas`

  5. **所有文档中将出现的「# ✅ 验证」命令**：逐条执行，记录真实输出作为文档预期输出
  6. **基本读写操作可用**：set/get、produce/consume、INSERT/SELECT 等

【全局写作风格】
- **简洁直接**：每句话必须有实质内容，禁止废话和套话（如"部署完成后，我们可以通过以下步骤..."）
- **一事一处**：同一内容只在最合适的章节出现一次，禁止跨章节重复描述
- **章节职责鲜明**：
  - 简介章节（第 1 章）：只讲"是什么、能解决什么问题"，不讲安装步骤
  - 生产环境规划（第 3 章）：只出现图/表/指标/角色说明，不出现任何 Shell 命令
  - 生产环境部署（第 4 章）：只写操作命令 + 必要说明，不再画图、不讲架构原理
  - 配置章节（第 5 章）：只讲参数含义，不重复部署步骤
  - 运维章节（第 7 章）：只写运维命令，不重复配置说明
- **特殊情况才说明**：若某步骤在大多数场景下都一样，不加额外解释；只有真正有坑、有差异时才补充说明
- **禁止**：过渡句（"接下来我们..."）、重复摘要（"如上所述..."）、显而易见的注释（`# 安装 nginx`）


【命令设计原则：顺序正确 + 幂等】
- **操作逻辑连贯性（顺序正确）**：
  - 严格遵循依赖顺序：任何操作步骤必须先满足前置条件，严禁逻辑倒置。
  - **先规划安装目录，再在规划目录下安装**（★ 强制执行）：
    1) 第 3.5 节规划好安装根目录及子目录结构（不出现 Shell 命令）
    2) 第 4.2 节第一步：按规划创建目录（`mkdir -p`）
    3) 第 4.2 节第二步：在已创建的目录下执行安装（解压、配置、启动）
    4) 第 4.5 节：安装完成后介绍实际目录结构及用途
    - 严禁在未规划目录的情况下直接安装到系统默认路径，或跳过目录创建步骤。
  - 先创建目录，后写入文件：在执行 `cat > /etc/xxx/conf` 之前，必须先有 `mkdir -p /etc/xxx` 步骤。
  - 先安装依赖，后配置服务：严禁配置未安装软件的配置文件（避免路径不存在）。
  - 先创建用户，后分配权限：严禁对不存在的用户执行 `chown/chmod`。
  - 先配置网络/防火墙，后启动服务：确保服务启动时环境已就绪。
  - 避免"回溯式"指令：不要在后续步骤中才补充说明"哦对了，之前那个步骤需要先做这个"。所有前置准备必须在主流程中按时间轴顺序提前完成。

- **命令幂等性规范（可重复执行）**：
  - 文档中所有 Shell 命令必须保证幂等性（即重复执行不报错、不产生副作用），
    以支持用户在测试环境中多次运行验证脚本。
  - 创建目录：使用 `mkdir -p`，而非 `mkdir`
  - 安装软件包：使用系统包管理器（dnf/apt），重复安装会自动跳过，无需额外判断
  - 写入配置文件：使用 `cat >` 覆盖写入，而非 `cat >>` 追加（避免内容重复）
  - 启动服务：使用 `systemctl enable --now`，而非 `systemctl start`（已运行时不报错）
  - 添加用户/角色：命令前加存在性判断，例如：
    ```bash
    id -u appuser &>/dev/null || useradd -r -s /sbin/nologin appuser
    ```
  - 下载文件：下载前检查文件是否已存在，例如：
    ```bash
    [ -f /tmp/package.tar.gz ] || wget -O /tmp/package.tar.gz "https://..."
    ```
  - 执行初始化操作（如数据库 init）：必须加已初始化判断，防止重复初始化损坏数据

  - 技术难点或易错点：使用 Markdown 引用块（`>`）加 ⚠️ 符号特别标注
  - 重要概念或推荐做法：使用 **加粗** 突出，并在括号内说明原因
  - 依赖官方文档的参数/行为：以 `[官方文档](链接)` 形式直接内联引用
  - 不同系统/版本存在差异时：在差异点前用 `> 📌 注意` 引用块说明版本差异

- 执行节点标注规范（每个操作步骤前必须明确节点）：
  ```
  > 🖥️ **执行节点：Master-01（192.168.x.x）**
  > 🖥️ **执行节点：所有节点（Master + Worker）**
  > 🖥️ **执行节点：任意一台 Master 节点**
  ```

- 每个关键操作步骤后，必须紧跟验证命令及预期输出，格式如下：

  # ✅ 验证
  systemctl status xxx
  # 预期输出：Active: active (running)
  ```
- 文档中所有「# ✅ 验证」命令必须在真实环境中执行过一次，确认预期输出无误后再写入文档。

【环境差异标注规范】

- 使用 🖥️ 符号标注"依赖环境类型"的参数，格式如下：

  ```yaml
  # 🖥️ 【环境差异】根据实际部署环境选择：
  #   - 物理机：直接使用物理网卡名（如 eth0/ens3），性能最优
  #   - 虚拟机（VirtualBox/VMware）：使用 eth1 或 ens33，注意网卡绑定
  #   - 云主机（阿里云/AWS/腾讯云）：使用内网 IP，禁用 IPIP 协议（云平台拦截）
  network_interface: "eth0"   # ★ ← 根据 ip addr 命令查到的实际网卡名填写
  ```

- 常见需要环境差异标注的参数场景（必须覆盖）：
  - **网卡名称 / 绑定 IP**：物理机固定名称、虚拟机可能是 eth1/ens33、云主机需查控制台
  - **存储路径 / 磁盘挂载**：物理机多块裸盘、虚拟机附加磁盘、云主机挂载数据盘路径不同
  - **网络模式（CNI/Calico 等）**：云主机推荐 VXLAN，物理机推荐 BGP，虚拟机均可
  - **内存 / CPU 限制参数**：根据实际节点规格调整，不应写死固定值
  - **时钟同步源（NTP）**：内网环境用内部 NTP 服务器，外网用公共 NTP
  - **防火墙策略**：云主机通常在安全组控制，不需要 firewalld；物理机需要

【容量规划规范（3.3 节必须遵循）】
- **目的**：帮助 SRE 在部署前合理评估资源需求，避免上线后因容量不足导致扩容或故障。
- **必须包含**（按服务特性选择适用项）：
  | 资源维度 | 估算方式 | 示例 |
  |---------|---------|------|
  | **磁盘** | 数据量 × 副本数 × 保留天数 × (1/压缩比) + 预留 30% | Kafka 日均 1TB，3 副本，保留 7 天 → 最低 21TB + 30% ≈ 28TB |
  | **内存** | 按服务特性：全内存型（Redis）/ JVM Heap（ES/Kafka）/ Page Cache / Buffer Pool | Redis 数据集 50GB → 单节点内存 ≥ 64GB（数据 + 碎片 + fork 开销） |
  | **CPU** | 按并发连接数、QPS、压缩/加密开销估算 | Kafka 启用压缩时 CPU 开销增加 30~50% |
  | **网络** | 峰值吞吐 × 副本同步系数 | Kafka 写入 500MB/s，3 副本 → 节点间带宽 ≥ 1.5GB/s |
- **写作要求**：
  - 提供估算公式或参考表，不要求精确计算，但必须给出量级参考
  - 按"小规模 / 中规模 / 大规模"三档给出推荐配置（表格形式）
  - 注明"以上为参考值，实际需根据业务压测结果调整"

【Systemd Unit 文件规范（生产部署必须遵循）】
- **适用范围**：所有通过二进制/源码方式部署的服务（非 Docker/K8s 管理的），必须提供 systemd unit 文件。
- **Unit 文件必须包含的关键参数**：
  | 参数 | 说明 | 推荐值 |
  |------|------|--------|
  | `User` / `Group` | 服务运行用户，禁止 root 运行数据服务 | 专用服务用户（如 redis、kafka、mysql） |
  | `LimitNOFILE` | 文件描述符上限，须与 4.1 节 ulimit 设置一致 | 65536 或更高 |
  | `LimitNPROC` | 进程数上限 | 65536 |
  | `Restart` | 异常退出后自动重启策略 | `on-failure` |
  | `RestartSec` | 重启间隔，避免快速循环重启 | `5s` |
  | `OOMScoreAdjust` | OOM 优先级调整，数据库类服务建议降低被 OOM kill 的概率 | `-900`（数据库）/ `-500`（消息队列） |
  | `TimeoutStartSec` / `TimeoutStopSec` | 启停超时，避免 systemd 过早 kill | 按服务启动耗时设置，通常 `60s`~`300s` |
- **创建格式**：使用 `cat > /etc/systemd/system/{service}.service << 'EOF'` 格式
- **创建后必须执行**：`systemctl daemon-reload && systemctl enable --now {service}`
- **验证**：`systemctl status {service}` 确认 Active: active (running)
- **示例模板**：
  ```ini
  [Unit]
  Description={ServiceName} Server
  After=network-online.target
  Wants=network-online.target

  [Service]
  Type=notify                          # 或 simple/forking，按服务实际选择
  User={service_user}
  Group={service_group}
  LimitNOFILE=65536
  LimitNPROC=65536
  OOMScoreAdjust=-900                  # 数据库类服务建议 -900
  Restart=on-failure
  RestartSec=5s
  TimeoutStartSec=120s
  TimeoutStopSec=120s
  ExecStart=/opt/{service}/bin/{binary} --config /opt/{service}/etc/{config}
  ExecStop=/bin/kill -SIGTERM $MAINPID
  WorkingDirectory=/opt/{service}

  [Install]
  WantedBy=multi-user.target
  ```

【日志清理与轮转规范（7.5 节必须遵循）】
- **目的**：防止日志占满磁盘导致服务异常，满足生产环境可追溯性要求；路径需与 4.5 节目录结构中的日志目录一致。

- **必须包含**：
  1. **方案选型**：根据服务特性选择最合适的方式（见下表），在章节开头 1~2 句话说明选型依据
  2. **保留策略表**：按日志类型（运行/审计/慢查询/GC 等）分别给出生产推荐值
     | 日志类型 | 保留天数 | 单文件上限 | 压缩 |
     |---------|---------|-----------|------|
     | 运行日志 | 7~14 天 | 100M~500M | 是 |
     | 审计/慢查询 | 30~90 天 | 100M | 是 |
  3. **可执行配置**：logrotate 配置 或 crontab + 清理脚本，必须可直接复制执行（`cat >` 格式）
  4. **部署步骤**：安装/放置配置 → 测试（`logrotate -d` 或脚本 dry-run）→ 加入 crontab（若用脚本）
  5. **验证方法**：配置生效后的验证命令及预期输出
  6. **磁盘监控建议**：建议对日志目录所在分区配置 Prometheus/告警（如使用率 > 85% 告警），便于提前干预

- **方案选型指引**：
  | 方式 | 适用场景 | 说明 |
  |------|----------|------|
  | **logrotate** | 服务使用文件句柄写日志，支持 copytruncate 或 restart | 优先推荐，系统原生、可压缩、可定时 |
  | **服务内置轮转** | 服务自身支持（如 MySQL slow_log、Redis log） | 在配置章节说明，7.5 节补充外部清理兜底 |
  | **crontab + 脚本** | logrotate 不适用或需自定义清理逻辑 | 提供完整脚本 + crontab 示例，脚本需幂等 |

- **生产环境标准要求**：
  - 保留天数：运行日志 7~14 天，审计/慢查询/错误日志 30~90 天（可追溯排障）
  - 轮转触发：按大小（如 100M）或按天，避免单文件过大影响 grep/分析
  - 压缩：轮转后旧日志 gzip 压缩，节省磁盘
  - ⚠️ **严禁**：直接 `rm` 正在写入的日志文件（会导致句柄丢失或服务异常）；使用 copytruncate 或信号触发重开
  - 脚本幂等：`find ... -mtime +N -delete` 类脚本可重复执行无副作用；crontab 注明执行用户（如 root 或服务用户）

- **写作格式示例**：
  - logrotate：`cat > /etc/logrotate.d/{service} << 'EOF'` 提供完整配置，含 `rotate`、`size`/`daily`、`compress`、`copytruncate`/`postrotate` 等
  - crontab + 脚本：脚本使用 `cat > /opt/{service}/scripts/cleanup-logs.sh << 'EOF'`，crontab 示例注明 `0 2 * * *` 等执行时机及 `chmod +x`
  - 多节点服务：注明「所有节点均需配置」，或给出 Ansible 等批量部署提示

- **差异说明**：若服务有多个日志类型（运行/审计/慢查询/GC 等），分别说明路径（与 4.5 一致）与推荐策略

- **7.5 节写作模板示例**（供参考，按实际服务调整）：
  ```markdown
  ## 7.5 日志清理与轮转

  本服务使用 **logrotate** 管理日志（推荐原因：系统原生、支持 copytruncate、可压缩）。所有节点均需配置。

  ### 保留策略
  | 日志类型 | 路径 | 保留天数 | 单文件上限 | 压缩 |
  |---------|------|---------|-----------|------|
  | 运行日志 | /data/xxx/log/*.log | 14 天 | 200M | 是 |
  | 慢查询   | /data/xxx/log/slow.log | 30 天 | 100M | 是 |

  ### logrotate 配置
  ```bash
  cat > /etc/logrotate.d/xxx << 'EOF'
  /data/xxx/log/*.log {
      daily
      rotate 14
      size 200M
      compress
      delaycompress
      copytruncate
      missingok
      notifempty
  }
  EOF
  ```

  ### 部署与验证
  ```bash
  # 测试配置（不实际执行）
  logrotate -d /etc/logrotate.d/xxx
  # 手动执行一次验证
  logrotate -f /etc/logrotate.d/xxx
  ```

  ### 磁盘监控建议
  建议对 `/data/xxx/log` 所在分区配置告警：使用率 > 85% 时告警，便于提前清理或扩容。
  ```

- **crontab + 脚本模板**（当 logrotate 不适用时）：
  ```markdown
  ### 清理脚本
  ```bash
  # 创建脚本目录，放置 cleanup-logs.sh（按 -mtime 删除 N 天前日志，脚本幂等）
  cat > /opt/xxx/scripts/cleanup-logs.sh << 'EOF'
  #!/bin/bash
  LOG_DIR="/data/xxx/log"
  find "$LOG_DIR" -name "*.log.*" -mtime +14 -delete
  find "$LOG_DIR" -name "*.gz" -mtime +14 -delete
  EOF
  chmod +x /opt/xxx/scripts/cleanup-logs.sh
  ```
  ### crontab
  ```bash
  # 每日 02:00 执行，以 root 或服务用户运行
  echo '0 2 * * * /opt/xxx/scripts/cleanup-logs.sh >> /var/log/xxx-cleanup.log 2>&1' | crontab -
  ```
  ```

- **Docker 部署说明**：若生产采用 Docker 部署，需在 7.5 中注明：容器 stdout/stderr 日志由 Docker 驱动管理（如 `json-file` 的 `max-size`/`max-file`），宿主机 logrotate 仅适用于挂载到宿主机的日志目录；二者分别说明。


【内核与系统级调优规范（4.1 节 Pre-flight Tuning，强制）】
- **原则**：不调优内核的中间件部署只能算「能跑」，不能算「生产级」。中间件性能往往取决于底层系统配置。
- **必须包含**：按服务特性列出该中间件依赖的**操作系统参数（Kernel Parameters）**，并提供可执行的调优命令及验证方法。
- **典型参数示例**（按服务实际选用，非全部罗列）：
  | 服务 | 参数/调优项 | 说明 |
  |------|-------------|------|
  | Redis | Transparent Huge Pages (THP) | 生产环境建议禁用，避免延迟抖动；`echo never > /sys/kernel/mm/transparent_hugepage/enabled` |
  | MySQL | fs.file-max、vm.swappiness | 提高文件句柄上限；降低 swap 倾向，避免数据库被换出 |
  | Elasticsearch | vm.max_map_count | 至少 262144，否则集群无法启动 |
  | Kafka | vm.swappiness、net.core.somaxconn | 降低 swap；提高连接队列 |
  | 通用 | ulimit (nofile、nproc) | 提高进程可打开文件数、进程数上限 |
- **写作要求**：每个参数须说明「作用 + 推荐值 + 验证命令（如 `sysctl -a \| grep xxx`、`cat /sys/kernel/mm/transparent_hugepage/enabled`）」
- **验证**：调优后必须执行验证命令确认参数已生效，并将验证命令及预期输出写入文档。


【常见故障排查与处理指南规范（10.2 节）】
- **必须按服务特性提供**：MySQL 主从不同步、Redis 集群节点 fail、Kafka 分区不可用、MongoDB 副本集选举异常、MySQL MGR 节点掉队重入等——根据该服务的典型生产故障场景编写
- **结构统一**：现象 → 原因分析 → 排查步骤（含命令）→ 解决方案
- **复杂故障须增加 Mermaid 状态流转图**：对于多状态、多分支的复杂故障（如 MGR 节点掉队→重连→恢复、Redis 集群 failover 流程、Kafka 分区 rebalance），除文字排查外，**必须**增加 Mermaid 状态机/流程图，直观展示状态流转与决策分支
- **处理命令须验证**：所有排查命令、修复命令必须在伪集群中模拟故障后实际执行，确认有效后再写入文档
- **示例格式**：
  ```markdown
  ### 故障：主从复制不同步
  **现象**：从库 `Seconds_Behind_Master` 持续增大或为 NULL
  **原因**：网络中断、主库 binlog 被 purge、从库 SQL 线程报错等
  **排查**：`SHOW SLAVE STATUS\G` 查看 Slave_IO_Running、Slave_SQL_Running、Last_Error
  **解决**：根据 Last_Error 处理（如跳过错误事务、重新做从库等），具体命令需在伪集群验证
  ```

  **复杂故障示例（含 Mermaid 状态图）**：在故障描述中插入 Mermaid 状态机图，格式如下：
  ```mermaid
  stateDiagram-v2
      [*] --> ONLINE
      ONLINE --> RECOVERING: 网络中断/宕机
      RECOVERING --> ONLINE: 自动追平
      RECOVERING --> OFFLINE: 落后过多/超时
      OFFLINE --> RECOVERING: 手动 rejoin
      OFFLINE --> [*]: 需重建
  ```
  图后紧跟「排查与解决」文字步骤。
