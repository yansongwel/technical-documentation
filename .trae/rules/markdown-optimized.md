【核心原则：先验证，后写文档】

编写任何中间件文档前，必须严格按以下顺序执行。禁止凭训练数据记忆直接输出文档。

**文档目标**：输出生产级多节点集群部署与运维文档（如 Redis 6 节点 Cluster、Kafka 3 Broker 集群、MySQL MGR 3 节点等），
文档中的架构、节点数、配置参数均按真实生产环境规模编写。

**验证目的**：在本机通过 Docker / Docker Compose 伪集群验证以下内容的正确性——
  - 下载地址是否可达
  - 安装命令是否存在且可执行
  - 配置参数是否正确且生效
  - 管理/运维命令是否可用且输出符合预期

**验证是手段，不是目的**：
  - 伪集群仅用于验证上述内容的正确性，验证通过后立即清理环境
  - 最终文档输出的是**生产级多节点集群方案**，而不是 Docker 伪集群方案
  - 伪集群的 Docker Compose 文件可作为第 6 章"快速体验部署"的内容保留

**强制执行顺序**：

```
第一步：版本探测（禁止凭记忆填写版本号）
  ↓
第二步：下载地址验证（实际请求确认可达性）
  ↓
第三步：Docker 伪集群安装与启动验证
  ↓
第四步：配置参数验证（在伪集群中应用生产级配置参数并确认生效）
  ↓
第五步：管理命令验证（逐条执行文档中将出现的所有命令，确认命令存在且输出符合预期）
  ↓
第六步：验证环境清理（删除容器、镜像、数据卷、临时下载文件，禁止遗留）
  ↓
第七步：将验证通过的内容转化为生产级多节点集群文档
```

【版本号获取规范（禁止凭记忆）】
- 禁止凭训练数据记忆填写版本号，必须通过实际命令查询后确认：
  ```bash
  # 方式1：查询 GitHub 最新 Release
  curl -s https://api.github.com/repos/{owner}/{repo}/releases/latest | grep '"tag_name"'

  # 方式2：查询 Docker Hub 最新稳定 Tag
  curl -s "https://hub.docker.com/v2/repositories/library/{image}/tags?page_size=10" | grep '"name"'

  # 方式3：包管理器查询可用版本
  dnf info {package}       # Rocky Linux
  apt-cache policy {package}  # Ubuntu

  # 方式4：官网下载页面 / Release Notes 页面确认（用 curl 请求验证 URL 可达）
  curl -sI "https://downloads.xxx.org/releases/{version}/" | head -1
  ```
- 所有版本号必须是"执行命令后得到的结果"，不得是"AI 认为应该是的版本"
- 若 GitHub API 有速率限制导致查询失败，改用 Docker Hub 或官网下载页确认

【Docker 伪集群验证规范】
- 验证环境：在当前 Linux 服务器上使用 Docker / Docker Compose 启动伪集群
- 验证目录统一放在 `/tmp/{service}-verify/`，便于集中清理
- 伪集群最低节点要求（仅用于验证，不写入生产文档主体）：
  - 数据库类（Redis/MySQL/MongoDB/ES）：最少 3 节点，验证集群选举与主从同步
  - 消息队列（Kafka/RabbitMQ/RocketMQ）：最少 3 Broker + 必要依赖组件
  - 配置中心（Nacos/Consul/etcd/ZooKeeper）：最少 3 节点，验证 Raft/ZAB 选举
  - 单机类服务（Nginx/Jenkins/Grafana）：单容器启动即可
- 验证内容必须覆盖：
  1. 下载地址可达性（wget/curl 实际请求，确认 HTTP 200）
  2. 容器/服务正常启动（无报错退出，`docker ps` 确认全部 running）
  3. 集群节点互通与状态检查（如 `cluster info`、`cluster nodes`、`show replicas`）
  4. 关键配置参数实际生效（通过命令查询确认，非仅启动成功）
  5. 所有文档中将出现的「# ✅ 验证」命令逐条执行，记录真实输出作为文档预期输出
  6. 基本读写操作可用（set/get、produce/consume、INSERT/SELECT 等）
  7. 文档中涉及的运维命令（备份、扩容、状态查看等）至少验证命令可执行

【验证后清理规范（强制执行）】
- 每个中间件验证完成并确认文档内容无误后，必须立即清理，禁止遗留验证环境：
  ```bash
  # 停止并删除所有验证容器及数据卷
  docker compose -f /tmp/{service}-verify/docker-compose.yml down -v 2>/dev/null

  # 删除临时下载文件与数据目录
  rm -rf /tmp/{service}-verify/

  # 清理未使用的 Docker 资源（悬空镜像、未使用网络等）
  docker system prune -f

  # 可选：删除验证用镜像以释放磁盘空间
  # docker rmi {image}:{tag}
  ```
- 清理完成后执行确认：
  ```bash
  # 确认无残留容器
  docker ps -a | grep {service} || echo "清理完成，无残留"
  # 确认临时目录已删除
  ls /tmp/{service}-verify/ 2>/dev/null || echo "临时目录已清理"
  ```

【核心原则：一个文档只讲一种方案】

当一个中间件存在多种生产级集群方案或多个主流版本时，禁止将所有方案/版本堆砌在同一个文档中。
必须拆分为多个独立文档，每个文档只聚焦一种方案，确保清晰、无干扰。

**拆分触发条件（满足任一条即拆分）**：
  1. **多种集群架构**：中间件提供多种官方支持的生产集群模式
     - 示例：Redis（Cluster 分片模式 vs Sentinel 哨兵模式）
     - 示例：MySQL（MGR 组复制 vs 主从复制 vs InnoDB Cluster）
     - 示例：Kafka（KRaft 模式 vs ZooKeeper 模式）
     - 示例：Elasticsearch（Hot-Warm-Cold 架构 vs 常规集群）
  2. **主流版本差异大**：两个大版本之间存在架构性变更、配置不兼容、或客户端/SDK 需要改造
     - 判断标准：如果从版本 A 升级到版本 B 需要修改业务代码或客户端配置，则视为"差异大"
     - 示例：Redis 7.x vs 8.x、Kafka 2.x(ZK) vs 3.x+(KRaft)、MySQL 5.7 vs 8.0

**文档组织结构**：

```
{service}/
├── {服务名}集群方案选型指南.md          ← 总览文档（必须）
├── {方案A 目录}/
│   └── {服务名}-{方案A}生产级部署与运维指南.md
├── {方案B 目录}/
│   └── {服务名}-{方案B}生产级部署与运维指南.md
└── ...
```

示例（Redis）：

```
redis/

├── redis-sentinel-production/
│   └── Redis-Sentinel生产级部署与运维指南.md

```

示例（MySQL 多版本 + 多方案）：

【选型总览文档规范（必须生成）】

  1. **方案对比表**：用表格对比所有可选方案的核心差异
     - 必须包含列：方案名称、架构模式、最低节点数、数据分片能力、自动 Failover、
       适用场景、复杂度、典型用户规模
  2. **版本对比表**（若涉及多版本）：
     - 必须包含列：版本号、发布日期、关键新特性、破坏性变更、客户端兼容性、
       是否需要改业务代码、大厂采用情况

【版本选择原则】

- **不盲目追新**：优先选择大厂广泛采用、社区成熟、生态完善的版本
- **生产稳定性优先**：若最新大版本发布不足 6 个月且无大厂背书，推荐上一个 LTS/稳定版
- **业务代码兼容性**：若新版本要求客户端 SDK 升级或业务代码改动，必须在选型文档中明确标注
- **双版本策略**：当两个大版本都有大量生产用户时（如 Redis 7.x 和 8.x），两个版本都提供部署文档
- 版本决策建议写在选型总览文档中，各部署文档的第 2 章仅引用总览文档，不重复

【执行流程】

编写某个中间件文档时，先评估是否需要拆分：
```
第零步：方案/版本评估
  ├─ 该中间件有几种生产级集群方案？
  ├─ 当前主流大版本之间差异是否足够大？
  ├─ 是否需要双版本部署文档？
  └─ 确定需要生成几个文档、分别是什么
  ↓
第一步：先生成选型总览文档（如需拆分）
  ↓
第二步：逐个生成各方案/版本的独立部署文档（每个文档走完整的验证流程）
```

【全局写作风格】
- **简洁直接**：每句话必须有实质内容，禁止废话和套话（如"部署完成后，我们可以通过以下步骤..."）
- **一事一处**：同一内容只在最合适的章节出现一次，禁止跨章节重复描述
- **章节职责鲜明**：
  - 简介章节（第 1 章）：只讲"是什么、能解决什么问题"，不讲安装步骤
  - 生产环境规划（第 3 章）：只出现图/表/指标/角色说明，不出现任何 Shell 命令
  - 生产环境部署（第 4 章）：只写操作命令 + 必要说明，不再画图、不讲架构原理
  - 配置章节（第 5 章）：只讲参数含义，不重复部署步骤
  - 运维章节（第 7 章）：只写运维命令，不重复配置说明
- **特殊情况才说明**：若某步骤在大多数场景下都一样，不加额外解释；只有真正有坑、有差异时才补充说明
- **禁止**：过渡句（"接下来我们..."）、重复摘要（"如上所述..."）、显而易见的注释（`# 安装 nginx`）


【命令设计原则：顺序正确 + 幂等】
- **操作逻辑连贯性（顺序正确）**：
  - 严格遵循依赖顺序：任何操作步骤必须先满足前置条件，严禁逻辑倒置。
  - 先创建目录，后写入文件：在执行 `cat > /etc/xxx/conf` 之前，必须先有 `mkdir -p /etc/xxx` 步骤。
  - 先安装依赖，后配置服务：严禁配置未安装软件的配置文件（避免路径不存在）。
  - 先创建用户，后分配权限：严禁对不存在的用户执行 `chown/chmod`。
  - 先配置网络/防火墙，后启动服务：确保服务启动时环境已就绪。
  - 避免"回溯式"指令：不要在后续步骤中才补充说明"哦对了，之前那个步骤需要先做这个"。所有前置准备必须在主流程中按时间轴顺序提前完成。

- **命令幂等性规范（可重复执行）**：
  - 文档中所有 Shell 命令必须保证幂等性（即重复执行不报错、不产生副作用），
    以支持用户在测试环境中多次运行验证脚本。
  - 创建目录：使用 `mkdir -p`，而非 `mkdir`
  - 安装软件包：使用系统包管理器（dnf/apt），重复安装会自动跳过，无需额外判断
  - 写入配置文件：使用 `cat >` 覆盖写入，而非 `cat >>` 追加（避免内容重复）
  - 启动服务：使用 `systemctl enable --now`，而非 `systemctl start`（已运行时不报错）
  - 添加用户/角色：命令前加存在性判断，例如：
    ```bash
    id -u appuser &>/dev/null || useradd -r -s /sbin/nologin appuser
    ```
  - 下载文件：下载前检查文件是否已存在，例如：
    ```bash
    [ -f /tmp/package.tar.gz ] || wget -O /tmp/package.tar.gz "https://..."
    ```
  - 执行初始化操作（如数据库 init）：必须加已初始化判断，防止重复初始化损坏数据

- 每个小章节中，遇到以下情况必须通过特定格式额外说明：
  - 技术难点或易错点：使用 Markdown 引用块（`>`）加 ⚠️ 符号特别标注
  - 重要概念或推荐做法：使用 **加粗** 突出，并在括号内说明原因
  - 依赖官方文档的参数/行为：以 `[官方文档](链接)` 形式直接内联引用
  - 不同系统/版本存在差异时：在差异点前用 `> 📌 注意` 引用块说明版本差异

- 执行节点标注规范（每个操作步骤前必须明确节点）：
  ```
  > 🖥️ **执行节点：Master-01（192.168.x.x）**
  > 🖥️ **执行节点：所有节点（Master + Worker）**
  > 🖥️ **执行节点：任意一台 Master 节点**
  ```

- 每个关键操作步骤后，必须紧跟验证命令及预期输出，格式如下：
  ```bash
  # ✅ 验证
  systemctl status xxx
  # 预期输出：Active: active (running)
  ```
- 文档中所有「# ✅ 验证」命令必须在真实环境中执行过一次，确认预期输出无误后再写入文档。

【操作系统兼容性】
- 文档以 **Rocky Linux 9 为主线**完整撰写所有部署步骤，无需为两个系统各写一遍完整流程
- Ubuntu 22.04 的处理原则：
  - **命令相同**（绝大多数配置、systemctl、内核参数等）：只写一份，不重复
  - **命令有差异**（包管理器、仓库配置、服务名等）：在该步骤下紧跟一个 Ubuntu 差异补充块

- Ubuntu 差异补充块格式（仅在有差异时出现，直接替换对应步骤，不重复完整章节）：

  ```bash
  # ── Rocky Linux 9 ──────────────────────────
  dnf install -y <package>

  # ── Ubuntu 22.04（差异）──────────────────── 
  # 其余步骤与 Rocky Linux 9 相同，仅以下命令不同：
  apt-get update && apt-get install -y <package>
  ```

- 禁止为 Ubuntu 单独开辟"4.3 Ubuntu 部署步骤"等平行大章节，
  除非两个系统的部署流程有根本性差异（如默认防火墙、cgroup 驱动不同）；
  此时才允许用独立小节，但必须注明"仅以下步骤与 Rocky Linux 9 不同"


【高可用架构标准】
- 生产部署章节默认按 3 节点或以上高可用/冗余架构编写
- Master/Primary 节点至少 2 个，保证故障切换


【配置文件创建格式】
- 创建新配置文件使用 `cat >`（覆盖写入），追加内容时再用 `cat >>`
- 所有配置文件的创建必须使用如下格式（方便终端直接粘贴执行）：

  ```bash
  cat > /etc/service/config.conf << 'EOF'
  # 配置内容（含注释）
  key = value   # ← ⚠️ 说明含义，用户根据实际修改
  EOF
  ```


【参数标注规范】
- 用 ★ 标注生产环境"必须修改"的关键参数
- 用 ⚠️ 标注"用户需结合自身环境判断"的参数
- 所有示例 IP、路径、密码均加上 `← 根据实际环境修改` 注释


【环境差异标注规范】
- 凡是配置文件中存在因部署环境不同（物理机 / 虚拟机 / 云主机）而导致参数值不同的项，
  必须用行内注释列出各场景的推荐值或注意事项，让读者自行判断选择。
- 使用 🖥️ 符号标注"依赖环境类型"的参数，格式如下：

  ```yaml
  # 🖥️ 【环境差异】根据实际部署环境选择：
  #   - 物理机：直接使用物理网卡名（如 eth0/ens3），性能最优
  #   - 虚拟机（VirtualBox/VMware）：使用 eth1 或 ens33，注意网卡绑定
  #   - 云主机（阿里云/AWS/腾讯云）：使用内网 IP，禁用 IPIP 协议（云平台拦截）
  network_interface: "eth0"   # ★ ← 根据 ip addr 命令查到的实际网卡名填写
  ```

- 常见需要环境差异标注的参数场景（必须覆盖）：
  - **网卡名称 / 绑定 IP**：物理机固定名称、虚拟机可能是 eth1/ens33、云主机需查控制台
  - **存储路径 / 磁盘挂载**：物理机多块裸盘、虚拟机附加磁盘、云主机挂载数据盘路径不同
  - **网络模式（CNI/Calico 等）**：云主机推荐 VXLAN，物理机推荐 BGP，虚拟机均可
  - **内存 / CPU 限制参数**：根据实际节点规格调整，不应写死固定值
  - **时钟同步源（NTP）**：内网环境用内部 NTP 服务器，外网用公共 NTP
  - **防火墙策略**：云主机通常在安全组控制，不需要 firewalld；物理机需要
